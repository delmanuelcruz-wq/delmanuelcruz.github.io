SQL Case Studies

The analytical layer behind the business impact.

Three short case studies showing how I'd express the revenue protection, data enrichment, and account-ops work from my portfolio in SQL — framed as problems a BI or analytics team would actually get asked to solve.

Queries written in ANSI SQL, compatible with PostgreSQL, BigQuery, or Snowflake. Results shown are representative.

Case 01 — Revenue Protection

Detecting international pricing variance across vendor price lists

At NEO, I caught a pricing discrepancy manually by comparing two international Nuura price list files side by side — a 53% European dealer discount applied to a US order that should have used the 50% contracted rate. Here's how that same catch scales as a repeatable SQL check.

The idea: join the vendor price list against the contracted dealer discount table, compute the expected vs. submitted unit price per order line, and flag any line where variance exceeds a tolerance band.

Turning a single manual catch into a recurring data quality check — so the next $263 vs. $247 discrepancy surfaces before the PO is ever processed. 
-- Flag order lines with pricing variance vs. contracted dealer discount
WITH expected_price AS (
  SELECT
    pl.sku,
    dc.dealer_id,
    dc.region,
    pl.list_price_usd,
    dc.contracted_discount_pct,
    ROUND(pl.list_price_usd * (1 - dc.contracted_discount_pct), 2) AS expected_unit_price
  FROM vendor_price_list pl
  JOIN dealer_contracts dc
    ON pl.vendor_id = dc.vendor_id
   AND dc.region = 'US'
)
SELECT
  o.order_id,
  o.customer_name,
  ol.sku,
  ol.submitted_unit_price,
  ep.expected_unit_price,
  ROUND(ol.submitted_unit_price - ep.expected_unit_price, 2) AS variance_usd,
  ROUND((ol.submitted_unit_price - ep.expected_unit_price)
        / ep.expected_unit_price * 100, 2) AS variance_pct,
  CASE
    WHEN ABS(ol.submitted_unit_price - ep.expected_unit_price) > 5
      THEN 'FLAG — review before entry'
    ELSE 'OK'
  END AS status
FROM orders o
JOIN order_lines ol ON ol.order_id = o.order_id
JOIN expected_price ep
  ON ep.sku = ol.sku
 AND ep.dealer_id = o.dealer_id
WHERE o.status = 'PENDING_ENTRY'
  AND ABS(ol.submitted_unit_price - ep.expected_unit_price) > 2
ORDER BY ABS(variance_pct) DESC;

Case 02 — Market Development

Scoring specifier firms by outreach readiness

The 833-firm specifier database isn't useful until it's prioritized. A sales or market-development team needs a single score that captures how actionable each firm is — contact depth, email verification, and LinkedIn presence — so outreach starts with the highest-value accounts first.

This query combines three weighted signals into a readiness score and segments firms into priority tiers.

Moving from "here's the list" to "here's the sequence, and here's why" — the difference between a data deliverable and a go-to-market asset. 
-- Prioritize specifier firms by outreach readiness
WITH firm_contacts AS (
  SELECT
    f.firm_id,
    COUNT(c.contact_id) AS contact_count,
    SUM(CASE WHEN c.email_verified THEN 1 ELSE 0 END) AS verified_emails,
    SUM(CASE WHEN c.linkedin_url IS NOT NULL
              THEN 1 ELSE 0 END) AS linkedin_hits
  FROM firms f
  LEFT JOIN contacts c ON c.firm_id = f.firm_id
  GROUP BY f.firm_id
)
SELECT
  f.firm_name,
  f.specifier_category,
  fc.contact_count,
  fc.verified_emails,
  fc.linkedin_hits,
  ROUND(
      (fc.verified_emails * 35)
    + (fc.linkedin_hits    * 25)
    + (LEAST(fc.contact_count, 5) * 8),
  0) AS readiness_score,
  CASE
    WHEN fc.verified_emails >= 2 AND fc.contact_count >= 3 THEN 'Tier 1 — Hot'
    WHEN fc.verified_emails >= 1 OR  fc.linkedin_hits   >= 3 THEN 'Tier 2 — Warm'
    ELSE 'Tier 3 — Needs Enrichment'
  END AS priority_tier
FROM firms f
JOIN firm_contacts fc ON fc.firm_id = f.firm_id
WHERE f.market = 'NYC'
ORDER BY readiness_score DESC;

Case 03 — Account Operations

Identifying at-risk accounts in a 600+ account portfolio

At Pharmsource, I managed 603 independent pharmacy accounts and coordinated every order across Purchasing, Accounting, and Sales. The operational question behind that coverage: which accounts are quietly slipping before they go fully dark?

This query flags accounts whose ordering cadence has materially degraded — comparing their last-90-day activity against their trailing baseline.

Account health is a leading indicator — the customers you rescue in week three are the customers you keep. Losing them in week eight is just slower math. 
-- Flag accounts whose recent order cadence has degraded materially
WITH recent AS (
  SELECT
    account_id,
    COUNT(*) AS orders_90d,
    SUM(amount_usd) AS revenue_90d
  FROM orders
  WHERE order_date >= CURRENT_DATE - INTERVAL '90 days'
  GROUP BY account_id
),
baseline AS (
  SELECT
    account_id,
    ROUND(COUNT(*) / 3.0, 1) AS avg_orders_per_90d,
    ROUND(SUM(amount_usd) / 3.0, 2) AS avg_revenue_per_90d
  FROM orders
  WHERE order_date BETWEEN CURRENT_DATE - INTERVAL '365 days'
                       AND CURRENT_DATE - INTERVAL '90 days'
  GROUP BY account_id
)
SELECT
  a.account_name,
  a.owner_rep,
  COALESCE(r.orders_90d, 0) AS orders_last_90d,
  b.avg_orders_per_90d   AS baseline_orders,
  COALESCE(r.revenue_90d, 0) AS revenue_last_90d,
  b.avg_revenue_per_90d  AS baseline_revenue,
  CASE
    WHEN COALESCE(r.orders_90d,0) = 0
         AND b.avg_orders_per_90d >= 2            THEN 'DARK — zero activity'
    WHEN COALESCE(r.orders_90d,0) < b.avg_orders_per_90d * 0.5
                                              THEN 'AT RISK — 50%+ drop'
    WHEN COALESCE(r.orders_90d,0) < b.avg_orders_per_90d * 0.8
                                              THEN 'Watch — 20%+ drop'
    ELSE 'Healthy'
  END AS health_flag
FROM accounts a
LEFT JOIN recent   r ON r.account_id = a.account_id
LEFT JOIN baseline b ON b.account_id = a.account_id
WHERE b.avg_orders_per_90d >= 1
ORDER BY
  CASE health_flag
    WHEN 'DARK — zero activity' THEN 1
    WHEN 'AT RISK — 50%+ drop'  THEN 2
    WHEN 'Watch — 20%+ drop'   THEN 3
    ELSE 4
  END,
  baseline_revenue DESC;
